# Investigation: Photo Slideshow 501 Error

**Date:** 2026-03-08
**Status:** RESOLVED ✅
**Severity:** Critical - Photo slideshow completely broken

## Summary

Photos stopped displaying on the kiosk TV, showing broken image icons. Investigation revealed that Apple iCloud's CDN requires HTTP/2 protocol for photo downloads. The issue was masked by URL testing with curl (which uses HTTP/1.1 by default), but the actual backend (Node.js fetch) and browsers work correctly with HTTP/2.

## Root Causes

### 1. .env File Missing (Fixed)
- **Problem:** `/var/www/kiosk/.env` was missing after deployment
- **Cause:** File is intentionally excluded from GitHub releases for security
- **Impact:** Backend couldn't load iCloud album URL from environment
- **Fix:**
  - Created .env file on Pi with `ICLOUD_ALBUM_URL`
  - Added validation to `auto-update.sh` to warn and attempt recovery from `config.public.json`

### 2. Derivative Selection Logic Regression (Fixed)
- **Problem:** March 7th commit (55f7e7eb) broke derivative selection with complex grouping logic
- **Original code:** Simple indexed approach `for (i = 0; i < items.length; i += 2)`
- **Broken code:** Attempted to group by photoGuid, but extraction logic was incorrect
- **Result:** Selected 228 derivatives instead of 114 photos
- **Fix:** Reverted to original simple indexed logic

### 3. HTTP/2 Protocol Requirement (Root Cause)
- **Problem:** Apple's iCloud CDN returns `HTTP 501 Not Implemented` for HTTP/1.1 requests
- **Discovery:**
  - curl (HTTP/1.1) → 501 error ✗
  - Node.js fetch (HTTP/2) → 200 OK ✓
  - Browser (HTTP/2) → 200 OK ✓
- **Impact:** Misleading during testing - URL tests with curl all failed, suggesting URLs were invalid
- **Reality:** URLs were always valid, just required HTTP/2

## URL Structure Analysis

### Working URL Example (Browser + Node.js fetch)
```
https://cvws.icloud-content.com/S/{checksum}/{filename}.JPG?
  o={signature}
  &v=1
  &z=https%3A%2F%2Fp143-content.icloud.com%3A443
  &x=1
  &a={auth_token}
  &e={expiry_timestamp}
  &r={request_id}
  &s={session_token}
```

### Parameters
- `o`: OAuth signature
- `v`: API version (always 1)
- `z`: Content delivery node URL
- `x`: Unknown (always 1)
- `a`: Authentication token (base64)
- `e`: Expiry timestamp (Unix epoch)
- `r`: Request UUID
- `s`: Session signature

### URL Generation
- Generated by iCloud `webasseturls` API endpoint
- Expiry: ~2 hours from generation
- Backend caches for 45 minutes (well before expiry)
- HTTP/2 required for access

## HTTP Protocol Comparison

| Client | Protocol | Result |
|--------|----------|--------|
| curl (default) | HTTP/1.1 | 501 Not Implemented ✗ |
| curl --http2 | HTTP/2 | 200 OK ✓ |
| Node.js fetch() | HTTP/2 | 200 OK ✓ |
| Browser (Chrome/Chromium) | HTTP/2 | 200 OK ✓ |
| Raspberry Pi Chromium 145 | HTTP/2 | 200 OK ✓ |

### 2. scripts/auto-update.sh
```bash
# Validate critical files after extraction
if [[ ! -f "$STAGING_DIR/.env" ]]; then
    log "WARNING: .env file missing after deployment"

    # Try to restore from config.public.json
    if [[ -f "$STAGING_DIR/server/data/config.public.json" ]]; then
        ALBUM_URL=$(jq -r '.photos.sharedAlbumUrl' "$STAGING_DIR/server/data/config.public.json")
        if [[ -n "$ALBUM_URL" && "$ALBUM_URL" != "null" ]]; then
            echo "ICLOUD_ALBUM_URL=$ALBUM_URL" > "$STAGING_DIR/.env"
            log "Created .env from config.public.json"
        fi
    fi
fi
```

## Lessons Learned

1. **Protocol Matters:** Always test with the same protocol the production system uses
2. **HTTP/2 Testing:** Use Node.js fetch or `curl --http2`, not plain curl
3. **Browser Cache:** Hard refresh required after backend fixes
4. **Derivative Selection:** Keep code simple - complex logic introduced bugs
5. **.env Validation:** Auto-deployment should validate critical files

## Prevention Measures

- **Auto-update script:** Added .env validation and recovery
- **Testing guide:** Document proper URL testing methodology
- **Deployment docs:** Emphasize .env file requirement
- **Code review:** Prefer simple solutions over complex logic
